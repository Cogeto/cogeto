# 0030 — Operator script and the pull-only deploy channel

**Status:** Accepted. **Context:** Session O6 (operator script) per
[`docs/Cogeto-v1-Roadmap-Revision.md`](../Cogeto-v1-Roadmap-Revision.md) D3:
operations are script-driven and manual-by-design — one good operator script,
run by hand on a fresh OVHcloud Ubuntu instance, handles install,
configuration, and upgrades, then prints a precise checklist of what the
operator must do (DNS records, OVH settings, verification steps). No
Terraform, no cloud-provider API automation, no self-serve provisioning, no
automatic updates. The operator-facing runbook is Unit B; this record freezes
the mechanism.

## Ruling 1 — A customer instance PULLS prebuilt signed images and never builds

The dev contract stays `docker compose up` on a fresh clone (§A.2), which
builds locally. A customer instance is the opposite: it must never build. Two
of the stack's containers existed only as local build targets, so the release
pipeline now publishes **three** images per `vX.Y.Z` tag, each amd64, each
signed with keyless cosign:

| Image | Contents |
| --- | --- |
| `cogeto/cogeto` | app / worker / migrate / preflight (Dockerfile target `runtime`, unchanged) |
| `cogeto/cogeto-edge` | Caddy edge with the built SPA baked in (target `caddy`) |
| `cogeto/cogeto-mail` | the receive-only Haraka inbound SMTP service (Session O4) |

The SBOM attestation stays on the main `cogeto/cogeto` image. CI's
`docker-build` job builds all three on every PR (no push) so none can rot.
Dev-only images (demo-seed, seed fixtures, the consoles edge, the redaction
sidecar) are deliberately **not** published — the first three are
sandbox/fixture tooling, and redaction is not part of the v1 deploy channel
(it can join it later with its own published image if a customer needs it).

## Ruling 2 — The deploy stack is a separate, fetched compose file

`project/infra/deploy/docker-compose.deploy.yml` + `Caddyfile` are the files
installed at `/srv/cogeto` on a customer instance, fetched from the release
tag that matches the pinned image version (plus the `zitadel-init/init.mjs`
bootstrap script). A separate file, not an override of the root compose,
because the two must be able to diverge structurally: the deploy stack has no
`build:` keys, marks every secret **required** (`${VAR:?}` — a missing .env
fails loudly; preflight/QS-8 stays the backstop), hardcodes
`COGETO_PRODUCTION=1`, always enables Qdrant API-key auth (QS-4), and carries
none of the dev profiles. Infra images keep the same digests as the dev stack
(QS-25); `project/src/entrypoints/operator-script.spec.ts` asserts all of
this plus digest/CSP consistency with the dev stack, so drift fails `test`.

TLS: the production Caddyfile serves `{$COGETO_EXTERNAL_DOMAIN}` with real
Let's Encrypt ACME (the proven pattern from the website deployment, moved
onto Caddy which the stack already uses). Caddy retries ACME until DNS
resolves, so install completes before DNS is pointed. The `s3.<domain>` vhost
is the browser-reachable presign origin (§A.9); the MinIO console and Qdrant
dashboard are **not** exposed on a customer instance.

## Ruling 3 — One script, checklist-first, refusing what it cannot verify

`scripts/operator/cogeto` (bash, shellcheck-clean, kept in the repo and
runnable standalone) with subcommands `install`, `configure`, `upgrade`,
`status`, `backup-info`, and a global `--check` dry run that validates
prerequisites and prints intended actions plus the checklist while mutating
nothing (that dry run is what CI exercises). Rules frozen here:

1. **Idempotent where possible; destructive never by default.** Every
   destructive or irreversible action requires typing an explicit
   confirmation phrase. `install` refuses an already-installed root; the
   script never deletes data.
2. **Secrets are generated locally** into `/srv/cogeto/.env` (mode 600):
   Postgres, MinIO root + KMS key, Zitadel masterkey/DB/admin, Qdrant API
   key, and the mail intake token. Names are logged, values never. The
   receipt-signing keypair is **not** generated by the script — it is born on
   first boot inside the `instance-keys` volume (decision 0008) and the
   script refuses to rotate it, loudly, because receipts chain to it (§B.1,
   decision 0026). Data-bound secrets (Postgres/MinIO/Zitadel passwords, the
   KMS key) are also refused by `configure --regenerate` — swapping the env
   alone bricks initialized state; only the mail intake token and the Qdrant
   API key rotate here.
3. **The inbound address is derived, not invented**: `capture@in.<domain>`
   with MX target `mail.<domain>` (decision 0028 ruling 1), written into
   `.env` for both the app and the mail service.
4. **Pinned known-good version.** The script defaults to `DEFAULT_VERSION`
   (bumped in each `chore: release` PR once verified) and refuses versions
   before `0.9.0` — the first release that publishes the edge/mail images.
   `upgrade` refuses unpublished tags (Docker Hub check), warns that
   migrations are forward-only on rollback, re-runs the migrate job via the
   compose dependency graph, and triggers `reindex` when memories record a
   different embedding model than configured (§A.4).
5. **Honest status.** `status` reports green only when everything it can
   observe works: container health, the app's aggregate `/api/health`
   (migrations, queue depth, dead-letter, sweep, encryption, mail), the
   actually-served TLS certificate, running-vs-configured image, disk, and
   `.env` permissions.
6. **Everything the script cannot do is a checklist.** Every run ends with a
   delimited "WHAT YOU MUST DO NOW" section with instance-specific values
   (real DNS records with the detected public IP, OVHcloud backup settings
   per D4, verification steps, vault items), grouped "do now" vs "verify
   after DNS propagates".

## Consequences

- The first installable release is the next one cut after this lands (the
  edge/mail images exist only from that tag forward). `DEFAULT_VERSION` in
  the script must be bumped as part of each release chore PR.
- Root-compose or dev-Caddyfile changes must be mirrored into
  `project/infra/deploy/` — the spec fails `test` on digest or CSP drift,
  and the deploy README states the rule.
- Backups remain an OVHcloud-panel concern (D4): `backup-info` prints what to
  enable; the script performs no backups and no restores.
