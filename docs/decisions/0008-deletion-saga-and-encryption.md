## Ruling 1 — MinIO encryption at rest: single-node KMS, static key (§A.9)

- The `minio` service runs with `MINIO_KMS_SECRET_KEY` (single-node KMS): one
  named key, format `<name>:<base64 of 32 bytes>`, generated **per instance**
  (`echo "cogeto-instance-key:$(openssl rand -base64 32)"` — documented in
  `.env.example`). The compose file bakes in a clearly marked DEV-ONLY default
  so a fresh clone still works with zero configuration (§A.2).
- `minio-init` enables **SSE-S3 default bucket encryption** (`mc encrypt set
  sse-s3`) and **asserts** the bucket reports it (shell `case` match on
  `mc encrypt info` — the `mc` image has no grep); failure fails compose up.
- The app's health check (`GET /api/health` → `minioEncryption`) re-asserts it
  for the instance's lifetime via a signed `GetBucketEncryption` call; the
  integration suite pins the same assertion (`bucket_encryption` test).
- **Key location:** the `.env` of the instance (compose-injected env var), never
  the repo or image. Losing it makes stored objects unreadable — it is backed up
  with the instance secrets.
- **Rotation caveat:** `MINIO_KMS_SECRET_KEY` is a static master key. Online
  rotation requires a KES deployment, which is deliberately **[Later]** — the
  v1 threat model (§A.9) is single-tenant isolation + SSE + disk encryption,
  and a KES sidecar is gold-plating today.

## Ruling 2 — Instance signing key: ed25519, generated by the migrate init job (§B.1)

- One ed25519 keypair per instance, generated **at first boot if absent** by the
  `migrate` init container (§A.2) into the dedicated `instance-keys` named
  volume (`COGETO_INSTANCE_KEY_DIR`, default `/instance-keys` in compose,
  gitignored `.instance-keys/` for bare local runs). Never in the repo or image.
- `migrate` is the only writer (volume mounted rw there, ro in app and worker;
  `instance-keys-init` hands the fresh volume to uid 1000 first). Private key
  `instance-signing-key.pem`, mode 0600; public `instance-signing-key.pub.pem`.
  The public half is written first so a crash between the two writes is
  recoverable, and re-derivable from the private key.
- `GET /api/instance/public-key` serves the public key **unauthenticated** —
  anyone holding an exported receipt can verify it independently.
- Node's built-in crypto does keygen/sign/verify; **no new dependency**.

## Ruling 3 — Object-storage access: a minimal SigV4 client, no SDK

The memory module owns all storage access for memory data (0003 ruling 2), now
including object bytes: `memory/persistence/object-store.ts` is a ~150-line S3
SigV4 client over `fetch` + `node:crypto` (put/delete/head/bucket-encryption,
path-style, MinIO-only assumptions fine for the single-tenant stack). The saga
needs five operations; that does not justify an SDK dependency (which would
need owner sign-off). Revisit only if O1 file upload outgrows it (multipart).

## Ruling 4 — Saga semantics: enumeration transaction + confirm-with-external-legs

- **Step 1 (app, one Postgres transaction):** enumerate derived memories by
  provenance `(source_type, source_id)` FOR UPDATE, null dangling
  `superseded_by` pointers of survivors (ruling 5), delete the memory rows,
  delete `file_metadata` + collect object keys for `file` sources, delete the
  source row through the **SourceDeletion port** (defined in memory,
  implemented in connectors, bound by the composition roots — the mirror of
  ingestion's SourceReader; keeps §A.1 rule 2 and the module graph acyclic),
  insert the receipt `pending` with `counts_json`, enqueue the worker job via
  the outbox, write the audit row. Any failure aborts everything.
- **Provenance completeness argument (why enumeration = the provenance query):**
  provenance is NOT NULL on every write path, and edit-supersession copies the
  predecessor's provenance onto the successor — so same-source chains are
  enumerated in full with no graph walk.
- **Authorization:** owner-only, checked against the source row (adapter /
  file_metadata) AND every derived memory row; mismatches surface as NotFound
  so existence does not leak.
- **Steps 2–3 (worker, one idempotent attempt):** delete Qdrant points (absent
  = success), delete MinIO objects (absent = success), then confirm the receipt
  with chain hash + signature **in the same idempotency transaction** keyed
  `(deletion_receipt, <receipt id>, deletion.execute)`. An external failure
  rolls the confirmation back, graphile retries with backoff, exhaustion parks
  the job in dead_letter with the receipt still `pending` — the receipt can
  never read `confirmed` while an enumerated identifier could still exist.

## Ruling 5 — Cross-source supersession chains: delete members, null pointers, record it

Deleting source S removes **only S's members** of any supersession chain. A
surviving memory (different source) whose `superseded_by` pointed at a deleted
row gets the pointer **nulled** (also required by the self-referencing FK), and
the receipt records those ids under `counts_json.superseded_by_nulled`. The
survivor keeps its own provenance and lifecycle status (`replaced` stays
`replaced`; its validity interval is already closed) — only the link to the
erased row disappears, because erasure of S must not be reconstructable from
what survives. Tested by `cross_source_chain`.

## Ruling 6 — Receipt chain: linkage is the order; advisory-lock serialization

- Canonical payload: `{id, source_type, source_id, counts_json, signed_at,
  confirmed_at, prev_hash}` → deterministic JSON (sorted keys at every depth,
  arrays ordered, JSON.stringify escaping, no unicode normalization) → SHA-256
  over the UTF-8 bytes → ed25519 signature over the hex hash. A golden-hash
  unit test pins the canonicalization forever.
- `prev_hash` links to the previous **confirmed** receipt; genesis constant
  `cogeto:deletion-receipt-chain:genesis`. Confirmation serializes on
  `pg_advisory_xact_lock(hashtext('cogeto_deletion_receipt_chain'))` and finds
  the tip as "the confirmed receipt no other confirmed receipt links to" —
  **linkage defines chain order, never timestamps**, so clock skew cannot fork
  or reorder the chain. More than one tip = corrupted chain = refuse to extend.
- `verifyChain()` (pure) walks genesis → tip recomputing hashes and verifying
  signatures; `GET /api/receipts/verify` exposes it. Migration **0009** adds
  the missing `signature` column (+ status/prev_hash indexes).
- `counts_json.requested_by` records the deleting principal **inside the signed
  payload**; a dedicated `owner_id` column waits for the Forgotten UI (F1-B) if
  list-filtering needs an index — single-tenant volumes do not yet.

## Ruling 7 — Dev seed fixture placement

`seedObjectFixture` lives in the memory module (it must write `file_metadata`,
module-private) and is called by the `seed-object` entrypoint script, which the
runtime image **deletes** (Dockerfile) and compose exposes only behind the
`dev-seed` profile running the build-stage image:
`docker compose --profile dev-seed run --rm seed-object --owner <id> --org <id>`.
